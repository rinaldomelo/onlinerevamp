# ADR-011 — Spec hierarchy as planner & architect input (by-reference)

- **Status:** Accepted
- **Date:** 2026-04-28
- **Deciders:** Rinaldo (solo dev)
- **Implements:** Milestone M14
- **Related:** ADR-005 (orchestrator stack), ADR-010 (planner ≠ architect split)

---

## Context

The L1 build-out (2026-04-28) shipped a 4-level project-wide spec hierarchy under `.claude/specs/` — `project.md`, `theme.md`, `pages/<slug>.md`, `sections/<slug>.md` — generated by the new `onboard-figma`, `init-specs`, `spec-project`, `spec-theme`, `spec-page`, `spec-section`, and `refresh-spec-viewer` skills. Those specs are loaded today by the **user-facing** skills (`/scope-feature`, `/plan-feature-implemenation`) but the **autonomous orchestrator's** planner and architect agents had no awareness of them.

M13 left `callPlannerModel` and `callArchitectModel` as scaffolds that throw `"not yet implemented at runtime"`; their TODO comments pointed at M14 for SDK wiring. The original M14 plan tried to bundle both — wire the runtime AND wire the spec hierarchy in one pass — but pivoted on the day of: the runtime wiring proved to be its own milestone (subprocess-based SDK auth, structured-output parsing, MCP wiring for two agents) that would have shipped untested in this environment. M14 instead scopes to the by-reference resolver + the contract for the eventual runtime; the runtime wiring is deferred to M15. The hero demo runs via in-conversation Agent sub-agents to validate the by-reference shape before M15 commits to it.

Two natural shapes:

- **By-value:** the harness pre-reads every relevant spec and inlines its markdown into the agent's system / user prompt before the call.
- **By-reference:** the harness emits a list of file paths into the user message; the agent reads what it needs via a tool. The agent decides which specs to actually consult.

By-value is simpler to reason about (no tool round-trips) but couples the resolver to prompt construction, makes prompts grow with the hierarchy (today: 4 sections; in 6 months: 30+), and forces every run to pay the token cost of every potentially-relevant spec — even when the model only needs `theme.md`.

By-reference matches how this kind of context is normally surfaced in agent SDKs (give the model a tool, let it pull what it needs), keeps the planner→architect bridge type (`TriagedFeatureRequestSchema` from ADR-010) untouched, and lets the resolver stay a small deterministic pure function with zero prompt-construction logic.

## Decision

**Specs are surfaced to the planner and architect by-reference.** The harness computes a list of repo-relative paths via `resolveProjectContext()` (`orchestrator/src/agents/shared/resolveProjectContext.ts`) and the agent reads what it needs via a scoped `read_spec` tool that allows reads only under `.claude/specs/` and `.claude/context/`.

Concretely:

- **Pure resolver** at `orchestrator/src/agents/shared/resolveProjectContext.ts` — ships in M14. Returns `ProjectContext` — repo-relative paths only.
- **Heuristic** (deterministic; no model in the loop):
  - `project.md`, `theme.md`, `OUTPUT-initial-theme-analysis.md`, `OUTPUT-initial-figma-analysis.md` always surfaced when present.
  - Page specs: exact match against `request.context.targetPages`, then title-substring fallback.
  - Section specs: union of (a) sections listed in matched pages' frontmatter, (b) a small alias map (`hero` → `slideshow`, `carousel` → `slideshow`, `banner` → `slideshow`), (c) word-boundary substring on the title.
  - Capped at 5 section matches to avoid prompt bloat from a sweeping title.
- **`read_spec` tool contract (implementation deferred to M15 — see "Open follow-ups").** Single arg `path: string`. Path-traversal guard limits reads to `.claude/specs/` or `.claude/context/`. Read-only, no write capability. The contract is captured here so M15 has a clear target; the M14 hero demo runs the planner+architect via in-conversation Agent sub-agents (which already have a `Read` tool) and proves the by-reference shape works at the prompt level.
- **Bridge contract preserved** — `TriagedFeatureRequestSchema` is unchanged. The resolver's output flows in as planner *input* only; the planner can cite spec paths in `references[]` per the existing `FeatureReferenceSchema`.
- **Architect path** — the architect runs the resolver again from its harness against the planner's already-emitted `triagedRequest.featureRequest`, getting the same set of paths. Idempotent and stateless — no need to pass paths through the bridge type.

## Consequences

**Positive:**

- **Prompt cost stays bounded** as the spec hierarchy grows. A future state with 50+ section specs costs the same per-run token-wise as today's 4 — only the path list grows, not the inlined markdown.
- **Agent decides what's relevant.** When the planner is reasoning about a hero tweak, it reads `slideshow.md` + `theme.md` and skips the other 3 section specs. When it's reasoning about a footer change, it reads different files. The resolver doesn't try to predict.
- **No coupling between resolver and prompt.** `prompt.md` is unchanged from M13. Resolver evolution doesn't require prompt edits.
- **Bridge contract preserved.** `TriagedFeatureRequestSchema` (ADR-010) didn't move. Architect's input still validates exactly as before. The only change at the harness layer is one extra call to `resolveProjectContext()`.

**Negative:**

- **Extra tool round-trips.** A planner run that consults 4 specs costs 4 tool calls (~roughly a second of latency each). For a feature with very simple context this is wasteful relative to by-value. Acceptable trade against the prompt-bloat alternative.
- **Resolver false positives.** Word-boundary substring on the title will match "footer" in a "header/footer audit" feature when only the header is wanted. Mitigated by the cap at 5 sections; the planner can simply ignore irrelevant matches.
- **Two harnesses both run the resolver.** Planner harness resolves once, architect harness resolves again from the same `featureRequest`. Idempotent and cheap, but it is two evaluations. If perf becomes an issue, plumb the resolver output through `TriagedFeatureRequestSchema.references[]` (already a `FeatureReferenceSchema[]`).

**Neutral:**

- **Reversible.** If by-reference proves wrong (e.g. the agent consistently reads all surfaced specs anyway), switching to by-value is a one-file change in each harness — pre-load the markdown and inline it in the user message. The resolver itself is unchanged.
- **Specialists unaware.** The liquid / config / assets / validation / deployment specialists don't see specs. This is intentional: specs are top-of-funnel context. The architect translates spec-grounded design into per-task work orders, and the specialists consume those work orders. Wiring specs into specialists is out of scope and not anticipated.

## Alternatives considered

| Alternative | Why rejected |
|---|---|
| **By-value: pre-load all relevant specs and inline** | Couples resolver to prompt construction. Prompt grows with hierarchy. Every run pays the full token cost regardless of what the agent actually needs. |
| **MCP server for `read_spec`** | Adds a process-boundary for one synchronous filesystem read. Over-engineered for the use case. |
| **Add `projectContext` to TriagedFeatureRequestSchema** | Breaks the M13 bridge contract. Architect would have to handle a non-empty / empty distinction. Resolver re-run from architect's own harness is simpler. |
| **One mega-prompt with the entire spec hierarchy inlined** | Would have worked at today's size (4 sections). Doesn't scale; once the project has 50+ sections, the prompt is unreadable. |

## Migration path

There is no migration *from* an older shape, only adding the resolver as input context. M14 ships:

1. `orchestrator/src/agents/shared/resolveProjectContext.ts` — pure function + `flattenContextPaths` helper + `parseSectionsFromFrontmatter`.
2. `tests/resolveProjectContext.test.ts` — covers the slideshow case + edge cases. Runs against the live `.claude/specs/` tree.
3. Figma MCP registration in `.mcp.json` for the autonomous-orchestrator runtime (the IDE has the Figma plugin; the autonomous orchestrator doesn't, even though M14 doesn't yet exercise it).
4. `milestones/M14-specs-as-planner-input.md` refreshed from stub to reflect what shipped.
5. The hero demo run via in-conversation Agent sub-agents — captures the planner's `TriagedFeatureRequest` and the architect's `Plan` + `Design` against the slideshow spec.

What M14 does NOT ship — these are M15 (orchestrator runtime wiring):

- `orchestrator/src/tools/read-spec.ts` — the actual MCP/SDK tool implementation.
- Wiring `callPlannerModel` and `callArchitectModel` to a real model runtime (Anthropic Agent SDK or Messages API). Both currently throw `"not yet implemented at runtime"`. M14 doesn't change that; the demo proves the by-reference shape via sub-agents instead.
- Augmenting `planner/harness.ts` / `architect/harness.ts` to call `resolveProjectContext()` and pass the result into the model call. The harness layer stays as-is for now; M15 plumbs the resolver in.

## Verification

- `pnpm test` runs the new resolver tests against the live `.claude/specs/` tree. If a spec file gets renamed, the relevant test fails with a clear pointer.
- Hero demo (Phase 3 of M14) — the captured planner output cites `slideshow.md` and `theme.md` in its `references[]`; the captured architect Plan + Design demonstrably grounds in those specs (e.g. references the slideshow spec's `## Notes` block on `FoxTheme.Carousel` reuse).
- `git diff main -- orchestrator/src/types.ts` shows zero changes — the M13 bridge contract is preserved.
- `git diff main -- orchestrator/src/agents/{planner,architect}/model.ts` shows only updated TODO comments — M14 doesn't activate the runtime.

## Open follow-ups (not blocking M14)

- **M15: orchestrator runtime wiring.** This is the big one — implement `read_spec` as either an SDK MCP tool or a thin native tool, wire `callPlannerModel`/`callArchitectModel` to a real model runtime, plumb the resolver through the harness. Open ADR-012 at that point capturing the SDK choice (Claude Agent SDK as a subprocess vs. raw Messages API). Until M15 lands, the orchestrator runs only via in-conversation sub-agents (the M14 demo path).
- **Figma MCP package canonicality.** The exact npm package for an autonomous Figma MCP isn't fully canon yet. M14 verifies before pinning; if no canonical package exists, document the gap and skip the wiring (the hero demo doesn't exercise Figma).
- **Resolver false positives.** Word-boundary substring is good enough today. If "false positive" becomes a recurring planner gripe in run logs, swap in a lemma-aware matcher or a small embedding lookup. Not premature.
- **Section-group matching.** `header-group` and `footer-group` are real spec slugs but the resolver currently only matches against `pages/<slug>.md` and `sections/<slug>.md`. When section-group specs land, extend the resolver to match them via the page's frontmatter (`group:` field).
- **Specs in the bridge type.** If the planner ends up needing to *commit* to a set of specs (e.g. for level-bound write-glob auditing), promote them into `TriagedFeatureRequestSchema.references[]` with `kind: "text"`. Right now the planner can already do this voluntarily; there's no enforcement.
- **Held-state spec citations.** When the planner holds, should `missingInputs` reference specific specs (e.g. "Confirm whether sections/slideshow.md's `autoplay_delay` covers your case")? Today it's free-form. Tightening is a future prompt edit, not a contract change.

## Findings

### 🔴 BLOCKERS (fix before merging the affected PR — or as the first follow-up commit)

#### 1. `orchestrator/tsconfig.json` — `rootDir: "./src"` + `include: ["tests/**/*"]` will fail compile
**Milestone:** M7
**Path:** `orchestrator/tsconfig.json:8` and `:18`
**Issue:** `rootDir` is `./src` but `include` adds `tests/**/*`. TypeScript will reject this with **TS6059** ("File 'tests/...' is not under 'rootDir' '<...>/src'") on `tsc --noEmit` (= `pnpm lint`) and on `tsc -p .` (= `pnpm build`). ADR-005 §Verification says "M7 ships with `pnpm install && pnpm test` running clean" — vitest will pass (it doesn't go through tsc), but `pnpm lint` and `pnpm build` will not. M10 spec line 52 explicitly requires "types compile."
**Fix:** Either drop `rootDir` (let TS infer from `include`), or split into `tsconfig.json` (src only) + `tsconfig.test.json` (extends, overrides include/rootDir for tests). One-line fix in either direction.

### 🟡 FIX-SOON (works but inconsistent — clean up in a follow-up commit on the affected branch)

#### 2. `dispatch.ts` validation/deployment cases reference M9/M10 but those milestones explicitly defer the wiring
**Milestone:** M8 (file shipped) / M9–M10 (architecture clarification)
**Path:** `orchestrator/src/orchestrator/dispatch.ts:21-23`
**Issue:** The switch arms return `notImplemented(task, "validation", "M9")` and `notImplemented(task, "deployment", "M10")`. But M9 spec §Out of scope and M10 spec §Out of scope both say "Wiring into the workflow runner happens after M11." So the message points to milestones that won't actually do the wiring. Architecturally, validation/deployment shouldn't be `PlanTask.targetAgent` values at all — they're invoked directly by the workflow runner *after* specialist dispatch — so the switch arms should return a clearer "should never receive this targetAgent" observation, not a "see M9" stub.
**Fix:** Update the notes to "validation/deployment do not run via dispatchTask; the workflow runner calls them directly post-specialists. PlanTask.targetAgent should never be 'validation' or 'deployment'." Optionally, throw instead of returning a stub observation, mirroring the `architect` arm at line 24-27.

#### 3. Specialist harnesses always log `planId: "unknown"` because they read from the wrong place
**Milestone:** M8
**Paths:**
- `orchestrator/src/agents/liquid/harness.ts:26, 44`
- `orchestrator/src/agents/config/harness.ts:25, 40`
- `orchestrator/src/agents/assets/harness.ts:23, 36`
**Issue:** Each harness reads `(payload as { planId?: string }).planId ?? "unknown"`. But `planId` lives on `Plan`, not on `PlanTask.payload` — `PlanTaskSchema` has `payload: z.record(z.unknown())` and the planner-architect doesn't populate planId into payloads. Every observation will record `planId: "unknown"`, breaking the M11 logger's "one file per plan" promise (`logger.ts:LOGS_ROOT/<planId>.jsonl`) — every run lands in `unknown.jsonl`. The smoke test at `tests/specialists.smoke.test.ts:18` masks this by manually injecting `planId: "P1"` into payload.
**Fix:** Either thread `planId` through `dispatchTask(task, planId)` (cleanest), or extend `PlanTaskSchema` with a top-level `planId: z.string()` field. The latter is minimally invasive; do that and have the planner-architect set it on every emitted task.

#### 4. `.gitignore` doesn't catch all `.env.*` files — secrets leak vector
**Milestone:** M3 (M3 owns `.env.example` + secrets baseline)
**Path:** `.gitignore:24-26`
**Issue:** Current entries are `.env`, `.env.local`, `.env.*.local`. Pattern `.env.*.local` only matches files like `.env.production.local`, not `.env.production`, `.env.staging`, `.env.dev`. A solo dev who copies `.env.example` to `.env.production` for prod testing will commit it. M11's `permissions.yml.global_forbidden` has `.env.*` (defense-in-depth at orchestrator-write time), but that doesn't help git.
**Fix:** Replace lines 24-26 with `.env*` then `!.env.example` to whitelist only the template. Two-line change.

#### 5. `deploy-dev-preview.yml` PR-comment template uses shell-variable syntax that won't expand
**Milestone:** M4
**Path:** `.claude/architecture/ci-staged/.github/workflows/deploy-dev-preview.yml:72`
**Issue:** Inside the `with: message: |` block (which is a YAML literal passed to the comment action, not a bash script), the line `Theme: \`$SHOPIFY_DEV_THEME_ID\`` will appear verbatim — the shell variable doesn't expand because this isn't a `run:` step. Result: every preview PR comment shows the literal string `$SHOPIFY_DEV_THEME_ID` instead of the theme ID.
**Fix:** Use GitHub Actions interpolation: `Theme: \`${{ secrets.SHOPIFY_DEV_THEME_ID }}\``. Note the secret will be masked as `***` in the comment, which is actually fine — the theme ID isn't security-sensitive but GitHub masks any registered secret. If unmasked theme ID is wanted, store the dev theme ID as a regular variable (`vars.SHOPIFY_DEV_THEME_ID`) rather than a secret.

#### 6. `deploy-dev-preview.yml` uses `--allow-live=false` (non-standard CLI flag form)
**Milestone:** M4
**Path:** `.claude/architecture/ci-staged/.github/workflows/deploy-dev-preview.yml:51`
**Issue:** Shopify CLI's `--allow-live` is a boolean flag; the canonical "no" is to omit it (default is `false`). Passing `--allow-live=false` may be silently misinterpreted as truthy by some flag parsers, depending on CLI version. The dev-preview workflow targets a non-live theme via `--theme $SHOPIFY_DEV_THEME_ID` so this is defense-in-depth, but it's still unreliable.
**Fix:** Drop the `--allow-live=false` line entirely. The default is already not-allow-live for non-live themes.

#### 7. All three workflows install `@shopify/theme` alongside `@shopify/cli` — redundant/legacy
**Milestone:** M3, M4, M5
**Paths:**
- `.claude/architecture/ci-staged/.github/workflows/theme-check.yml:40`
- `.claude/architecture/ci-staged/.github/workflows/deploy-dev-preview.yml:43`
- `.claude/architecture/ci-staged/.github/workflows/deploy-production.yml:30, 62`
**Issue:** `npm install -g @shopify/cli @shopify/theme` — the `@shopify/theme` package was the legacy theme CLI; current Shopify CLI (v3.x+) ships theme commands inside `@shopify/cli`. Installing both either duplicates or, on some npm registry states, errors. At minimum, it slows CI by ~20s per run for no benefit.
**Fix:** Strike `@shopify/theme` from all three install lines. Use `npm install -g @shopify/cli` only.

#### 8. `deploy-production.yml` step summary leaks masked secret as `***`
**Milestone:** M5
**Path:** `.claude/architecture/ci-staged/.github/workflows/deploy-production.yml:79`
**Issue:** `echo "- **Theme ID:** \`${{ secrets.SHOPIFY_PROD_THEME_ID }}\`"` writes the secret to `$GITHUB_STEP_SUMMARY`. GitHub Actions auto-masks any registered secret in any output, so the summary will display `Theme ID: ***`. The line is therefore useless — anyone reading the deploy log can't see which theme was pushed.
**Fix:** Either (a) move `SHOPIFY_PROD_THEME_ID` from secrets to repository variables (`vars.SHOPIFY_PROD_THEME_ID`) — theme IDs aren't credentials, just identifiers — or (b) delete this line from the summary.

#### 9. `format:check` covers all `.md` files but excludes `.js`
**Milestone:** M2
**Path:** `package.json:13`
**Issue:** `prettier --check "**/*.{json,liquid,css,md,yml,yaml}"` — covers `.md` (so all 11 milestone specs, 8 ADRs, runbooks, etc. must pass prettier formatting), but skips `.js`. Two consequences: (a) any markdown file in `.claude/architecture/` that wasn't prettier-formatted at write time will fail format:check on first CI run (this includes specs that may use prettier-incompatible markdown like wide tables); (b) the theme's actual JS assets are never format-checked, which contradicts the goal of consistent formatting.
**Fix:** Run `prettier --check "**/*.{json,liquid,css,md,yml,yaml}"` locally now to surface any pre-existing formatting violations and either prettier-format them or extend the prettierignore. Consider adding `.js` to the glob — but only after auditing the existing assets/*.js files for compatibility.

**Path:** `orchestrator/src/tools/shopify-dev-mcp.ts:7`

#### 11. `tools/fs.ts:53-60` — `listDir` is non-recursive
**Path:** `orchestrator/src/tools/fs.ts:53`
**Issue:** `listThemeFiles` reports only top-level entries per directory. For `assets/` and `templates/customers/`, nested files aren't enumerated. The planner-architect's input shape only needs sections/snippets/templates lists (which are flat in Shopify themes), so this works today. But if a future milestone wants the full theme map, this lies.
**Fix:** Use `readdir(path, { recursive: true })` (Node 20+). One-line change.

#### 12. `cache: 'npm'` without a lockfile in repo
**Paths:** all three staged workflows under `setup-node@v4` steps
**Issue:** No `package-lock.json` is committed (M2 ships only `package.json`). `actions/setup-node` will warn "no package-lock.json found, npm cache will not be effective." Cosmetic — installs still work.
**Fix:** Either remove `cache: 'npm'` from the action inputs, or commit a lockfile after running `npm install` once locally.

#### 13. `shopify.theme.toml` `[environments.production]` while branch is `main` — naming asymmetry
**Path:** `shopify.theme.toml:29`
**Issue:** The git workflow uses `main` as the production branch, but the toml environment is named `production`. Sensible mapping (branch → env-name aren't 1:1 by design), but the deploy-production workflow could be more explicit: comment that `--environment=production` maps to the `main` branch's deploy target.
**Fix:** Add a one-line comment in the workflow YAML or in `shopify.theme.toml` documenting the mapping. Not urgent.

#### 14. M11 `policy.ts` inline YAML parser is acknowledged-fragile but works for current `permissions.yml`
**Path:** `orchestrator/src/orchestrator/policy.ts:100-156`
**Issue:** Verified by hand-tracing every line of `permissions.yml` — the parser handles all current shapes correctly: top-level `agents:` / `global_forbidden:`, 2-space indented agent names, 4-space indented field keys (`write_globs:`, `read_globs:`, `forbidden:`), inline `[]` empty arrays, and 6-space `- 'value'` list items with comments stripped. It will silently mis-parse if any nested structure or unquoted multi-line value is added. M11 spec already lists "replace with js-yaml" as a known follow-up.
**Fix:** Already on the M11 close-out checklist — no new action needed.
